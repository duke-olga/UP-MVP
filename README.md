# UP-MVP

MVP модуля формирования учебного плана для системы «Интеллектуальный методист».

Текущая версия уже учитывает изменения из `source/tz.md`:
- в Таблице 1 есть отдельные блоки обязательных элементов ФГОС;
- практики хранят обязательный атрибут `practice_type`;
- в Таблице 2 поддерживаются дополнительные часы для дисциплины «Физическая культура и спорт»;
- в Таблице 3 нормативные расчёты строятся только по данным Таблицы 2.

## Запуск с нуля

1. Создать и активировать виртуальное окружение Python.
2. Установить backend-зависимости из `requirements.txt`.
3. Убедиться, что локально установлен и запущен Ollama.
4. Установить frontend-зависимости в `frontend/`.
5. Запустить backend на `http://127.0.0.1:8000`.
6. Запустить frontend на `http://127.0.0.1:5173`.

## Backend

Требования:
- Python 3.11+

Установка зависимостей:

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

Запуск backend:

```bash
uvicorn backend.main:app --reload
```

Проверка health-check:

```bash
curl http://127.0.0.1:8000/api/v1/health
```

Ожидаемый ответ:

```json
{"status":"ok"}
```

## Ollama

Проверка установки и доступных моделей:

```bash
ollama --version
ollama list
```

В текущем проекте по умолчанию используется модель `llama3:latest`.
Если хочешь использовать другую локальную модель, задай переменную окружения `OLLAMA_MODEL`.
При медленном ответе локальной модели можно увеличить таймаут через `OLLAMA_TIMEOUT`.

Пример для PowerShell:

```powershell
$env:OLLAMA_MODEL="llama3:latest"
$env:OLLAMA_TIMEOUT="180"
uvicorn backend.main:app --reload
```

Если Ollama уже запущен как Windows-приложение, отдельная команда `ollama serve` не нужна.

Быстрая проверка LLM через API проекта:

1. Запустить backend.
2. Создать план.
3. Добавить хотя бы один элемент в Таблицу 2.
4. Вызвать `POST /api/v1/plans/{plan_id}/validate`.
5. Убедиться, что в ответе заполнено поле `llm_recommendations`.

## Frontend

Установка зависимостей:

```bash
cd frontend
npm install
```

Запуск dev-сервера:

```bash
cd frontend
npm run dev
```

Frontend ожидает backend по адресу `http://localhost:8000` через proxy Vite.

Что уже есть в skeleton:
- React 18 + Vite;
- proxy `/api` -> `http://localhost:8000`;
- `src/api.js` с базовым axios-инстансом;
- страницы `Table1`, `Table2`, `Table3`;
- базовая проверка backend через `/api/v1/health`.

Что уже реализовано в рабочих экранах:
- список планов и создание нового плана;
- Таблица 1 с отдельными блоками:
  - обязательные дисциплины ФГОС;
  - обязательные категории практик ФГОС;
  - рекомендации по компетенциям;
  - ручной режим для ПКС;
- Таблица 2 с добавлением, редактированием и удалением строк, выбором `practice_type` и учётом дополнительных часов;
- Таблица 3 с агрегатами, отклонениями, нарушениями, LLM-рекомендациями, утверждением и экспортом XLSX.

## Что важно в предметной модели

- `competencies.json` моделирует входной перечень компетенций из ОПОП ТИУ.
- Автоматические рекомендации формируются для `УК`, `ОПК` и `ПК`, если `ПК` реально присутствуют во входном списке.
- Для `ПКС` автоматические рекомендации не формируются: пользователь работает вручную в Таблице 2.
- Для практик используется атрибут `practice_type = educational | industrial`.
- Для физкультуры общий объём программы в з.е. считает только `credits`, а дополнительные часы учитываются отдельно в суммарной нагрузке.

## Демо-сценарий

1. Открыть frontend.
2. Создать новый план.
3. Перейти в Таблицу 1 и выполнить перенос рекомендаций в Таблицу 2.
4. В Таблице 1 при необходимости выбрать варианты по обязательным дисциплинам и практикам ФГОС.
5. В Таблице 2 при необходимости вручную добавить элементы для ПКС, недостающих блоков или обязательных элементов, которые не были выбраны в Таблице 1.
6. Для практик задать `practice_type`, а для физкультуры при необходимости указать дополнительные часы.
7. Перейти в Таблицу 3 и нажать «Проверить учебный план».
8. Посмотреть список нарушений и текст `llm_recommendations`.
9. Скачать XLSX.
10. Попробовать утвердить план.
11. Если есть критические ошибки или ошибки, утверждение будет заблокировано.
12. После исправления структуры плана повторно выполнить проверку и утвердить план.

## Импорт PDF-источников в seed JSON

- Входные PDF ПООП хранятся в `backend/seed/poop_pdf/`.
- PDF лучших практик хранятся в `backend/seed/best_practices_pdf/`.
- Нормализованный JSON сохраняется в `backend/seed/poop_disciplines.json`.
- Отчет качества импорта сохраняется в `backend/seed/poop_import_report.json`.
- Импорт использует универсальный pipeline с тремя extractor-слоями: сначала `PyMuPDF`, затем `Docling`, затем `pypdf` как текстовый fallback. Поверх extractor-ов строятся candidate rows, детерминированная нормализация, quality gates и при необходимости fallback через локальную LLM в Ollama.
- На плохих или агрегированных строках импорт не падает: пишет предупреждение в лог и идет дальше.
- Доступны три режима:
  - `deterministic` — только правила и парсинг таблиц;
  - `hybrid` — сначала правила, потом fallback через локальную LLM в Ollama;
  - `llm` — только LLM-извлечение по релевантным табличным блокам.
- Для лучших практик `source_type` записывается как `best_practices`.
- Для ПООП `source_type` записывается как `poop`.
- В report-файле для каждого PDF сохраняются `extractor_used`, `candidate_row_count`, `record_count`, `quality_score`, `needs_review` и примеры candidate rows.
- `hybrid` теперь включает LLM не только при `0` записей, но и при слабом качестве детерминированного результата.

Запуск:

```bash
python scripts/import_poop_pdf.py
```

Или с явными путями:

```bash
python -m backend.modules.seed_ingest.poop_pdf_importer --poop-dir backend/seed/poop_pdf --best-practices-dir backend/seed/best_practices_pdf --output backend/seed/poop_disciplines.json --report backend/seed/poop_import_report.json
```

Рекомендуемый AI-assisted режим:

```bash
python scripts/import_poop_pdf.py --strategy hybrid
```

Для `hybrid` и `llm` должен быть локально запущен Ollama. Если Ollama недоступен, `hybrid` не падает и оставляет только детерминированный результат.

Практически для разнородных ПООП и учебных планов других вузов лучше использовать `hybrid`.
Для локального Ollama под эту задачу разумно начинать с моделей класса `qwen2.5:7b-instruct` или сильнее, если хватает памяти.
Для fallback через `Docling` и `pypdf` библиотеки должны быть установлены локально из `requirements.txt`. Если один из extractor-ов недоступен, pipeline не падает и идет дальше по остальным слоям.
