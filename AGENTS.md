# AGENTS.md
# Постоянные правила проекта — Модуль формирования учебного плана

---

## Контекст

Веб-модуль «Формирование учебного плана» — часть системы «Интеллектуальный методист».
Логически продолжает модуль ОПОП. В MVP симулирует получение компетенций из ОПОП через seed-данные.
Основной пользователь — методист. Система поддерживает, но не заменяет его решения.

---

## Стек (фиксированный для MVP)

| Слой | Технология |
|---|---|
| Backend | FastAPI (Python 3.11+) |
| ORM / валидация | SQLAlchemy + Pydantic v2 |
| БД | SQLite (один файл, без сервера) |
| Frontend | React 18 + Vite |
| HTTP-клиент (фронт) | axios |
| Экспорт | openpyxl |
| LLM | Ollama (локальный сервер, порт 11434) |

Никаких платных API. Никакого Docker как обязательного требования MVP. Всё запускается локально на ноутбуке.

## Архитектура: modular monolith

### Backend — внутренние модули

```
backend/
├── main.py                  # точка входа FastAPI, регистрация роутеров
├── database.py              # подключение SQLite, сессия
├── models.py                # SQLAlchemy ORM-модели
├── schemas.py               # Pydantic-схемы запросов/ответов
├── modules/
│   ├── seed_ingest/         # загрузка начальных данных из JSON
│   ├── recommendation/      # логика Таблицы 1: сопоставление компетенций и элементов
│   ├── plan_builder/        # логика Таблицы 2: расчёты, редактирование структуры УП
│   ├── validation/          # нормативные проверки (детерминированные, без LLM)
│   ├── llm_explainer/       # вызов Ollama, формирование промпта, получение рекомендаций
│   └── export/              # генерация XLSX
├── routers/
│   ├── competencies.py
│   ├── table1.py
│   ├── table2.py
│   ├── table3.py
│   └── export.py
└── seed/
    ├── competencies.json
    ├── poop_disciplines.json
    └── normative_params.json
```

### Адаптеры (принцип расширяемости)

Каждый модуль взаимодействует с внешними зависимостями через адаптеры:
- `db_adapter` — обёртка над SQLAlchemy (в будущем: замена на PostgreSQL без изменения логики)
- `llm_adapter` — обёртка над Ollama (в будущем: замена на другую LLM без изменения llm_explainer)
- `data_source_adapter` — источник нормативных параметров и seed-данных (в будущем: подключение к API модуля ОПОП)
- `auth_adapter` — заглушка, возвращает одного пользователя (в будущем: JWT/OIDC без переделки бизнес-логики)

## Принципы расчётов

1. **Часы = з.е. × CreditHourRatio (по умолчанию 36).** Рассчитываются автоматически в `plan_builder`. Поле `hours` никогда не принимает ввод пользователя.
2. **Все нормативные проверки — детерминированный Python-код** в модуле `validation`. Результат одинаков при одних и тех же входных данных.
3. **Нормативные параметры** (X_total, X_b1, X_b2, X_b3, X_year, X_mandatory_percent, X_pe_ze, X_pe_hours, X_semester_max, CreditHourRatio) поступают из ОПОП. В MVP хранятся в таблице `NormativeParams`, загружаются из `normative_params.json` при старте. ФГОС напрямую в этом модуле не разбирается.
4. **Агрегаты Таблицы 3** пересчитываются автоматически при каждом изменении Таблицы 2.

---

## Роль LLM (жёсткие правила)

- LLM подключается **только после** завершения всех детерминированных проверок.
- LLM получает **только структурированный отчёт** о нарушениях (список объектов: rule_id, level, message, actual, expected).
- LLM **не выполняет нормативные расчёты**, не обращается к БД напрямую, не изменяет план автоматически.
- Рекомендации LLM носят консультативный характер и **не блокируют и не разблокируют утверждение**.
- Промпт формируется в `llm_explainer` и включает: системный контекст, список нарушений, запрос на пояснения и предложения по корректировке.
- В MVP — без RAG, без векторного поиска.

## Логика таблиц (строго по спецификации)

**Таблица 1** — витрина рекомендаций:
- компетенции подтягиваются из ОПОП, **не редактируются**;
- пользователь **не добавляет и не удаляет** дисциплины/практики — это список рекомендаций системы;
- для вариативных дисциплин — только чекбокс «Включить в УП»;
- обязательные позиции включаются автоматически;
- **вариативные практики в MVP не предлагаются в Таблице 1** — добавляются вручную в Таблице 2;
- Таблица 1 **не участвует в расчётах и нормативных проверках**.

**Таблица 2** — основная рабочая таблица УП:
- заполняется на основе переноса из Таблицы 1 + ручные дополнения методиста;
- все расчёты, проверки, агрегаты — здесь.

**Таблица 3** — автоматический контроль:
- строится только на данных Таблицы 2;
- обновляется при каждом изменении Таблицы 2;
- показывает отклонения з.е. по блокам/годам и процентные отклонения.

## ПК и ПКС

- Автоматическое сопоставление безопасно только для **УК и ОПК** (совпадают в ФГОС и ПООП).
- Для **ПК** — осторожное сопоставление, только если вуз принял ПК из ПООП.
- Для **ПКС** (собственные компетенции вуза) — **ручной режим в MVP**: пользователь добавляет дисциплины в Таблицу 2 вручную, автоподбор не применяется.
- Код не должен делать автосопоставление для ПКС без явного флага источника.

## Подготовка к облачному режиму (не реализуется, но учитывается в архитектуре)

- Все зависимости (БД, LLM, источник данных, авторизация) скрыты за адаптерами.
- SQLite → PostgreSQL: замена строки подключения в `database.py`.
- Ollama → внешний LLM: замена `llm_adapter`.
- `auth_adapter` → JWT/OIDC без изменения бизнес-логики.
- `data_source_adapter` → API модуля ОПОП без изменения `recommendation/`.
- CORS настраивается через переменную окружения `ALLOWED_ORIGINS` в `main.py`.

## Соглашения по коду

- Все API-маршруты: REST, префикс `/api/v1/`
- Ответ успеха: `{ "data": ... }`, ответ ошибки: `{ "detail": "..." }`
- Pydantic-схемы обязательны для всех входящих и исходящих данных
- Фронтенд: все запросы только через `src/api.js`
- `hours` — только вычисляемое поле, никогда не принимается из запроса пользователя
