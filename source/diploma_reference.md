## ГЛАВА 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ

### 1.1 Общая характеристика бизнес-процесса объекта исследования

#### Основные виды деятельности объекта цифровизации

Объектом цифровизации является учебно-методический отдел (УМО) вуза, в частности — процесс разработки и актуализации образовательных программ и учебных планов.

Основные виды деятельности:

1. **Разработка основной профессиональной образовательной программы (ОПОП)** — определение компетенций выпускника, целей и результатов подготовки, нормативного срока и форм освоения программы. Регулируется ФГОС ВО 3++ соответствующего направления подготовки.

2. **Формирование учебного плана (УП)** — структурирование дисциплин, практик, ГИА и факультативов по блокам и семестрам, распределение трудоёмкости (зачётных единиц), установление связи каждого элемента с компетенциями.

3. **Нормативная проверка** — контроль соответствия сформированного учебного плана требованиям ФГОС: проверка общего объёма (240 з.е.), нагрузки по блокам, годам и семестрам, наличия обязательных дисциплин, покрытия компетенций.

4. **Согласование и утверждение** — процедура рассмотрения и подписания УП на уровне кафедры, методического совета и учёного совета вуза.

5. **Актуализация** — переработка действующих программ при изменении нормативной базы (новая редакция ФГОС, изменение профессиональных стандартов), а также плановое обновление раз в 1–2 года.

6. **Разработка рабочих программ дисциплин (РПД)** — нижестоящий уровень документации, формируемый на основе утверждённого УП.

#### Организационная структура объекта цифровизации

```
Учёный совет вуза
    ├── Методический совет / УМО
    │       ├── Методист (основной пользователь системы)
    │       └── Методический отдел кафедр
    └── Выпускающая кафедра
            ├── Заведующий кафедрой
            └── Преподаватели — разработчики дисциплин
```

**Методист** выполняет ключевую роль: координирует разработку, обеспечивает нормативное соответствие, взаимодействует с кафедрами. Именно методист является **основным пользователем** проектируемой системы.

Информационные потоки в бизнес-процессе:
- **Вход:** требования ФГОС ВО, профессиональные стандарты, ПООП (примерные образовательные программы), учебные планы других вузов, внутренние нормативные документы (положения, регламенты).
- **Выход:** ОПОП (перечень компетенций), учебный план в формах XLSX/PDF, отчёты о нормативных проверках.
- **Управление:** законодательство об образовании, ФГОС, Порядок разработки ООП вуза.
- **Механизмы:** методист, АИС/ИС вуза (если имеется), офисные приложения.

### 1.2 Проблемы, актуальность и обоснование проектного решения

#### Ключевые проблемы существующего процесса

| Проблема | Следствие |
|----------|-----------|
| Ручная проверка соответствия нормативам ФГОС | Ошибки в расчётах з.е., пропущенные дисциплины, нарушения структуры блоков |
| Разрозненность источников данных (ПООП, планы других вузов, внутренние документы в разных форматах) | Трудоёмкое сопоставление; субъективность при подборе дисциплин |
| Отсутствие инструмента сопоставления дисциплин и компетенций | Неполное покрытие компетенций; зависимость от опыта конкретного методиста |
| Каждая актуализация — повторная ручная работа с нуля | Большие трудозатраты при каждом обновлении ФГОС |
| Нет единого формализованного источника знаний по нормативным требованиям | Разные специалисты интерпретируют ФГОС по-разному |

#### Актуальность

Цифровая трансформация образования закреплена в Национальной стратегии развития ИИ до 2030 года (Указ Президента РФ от 10.10.2019 № 490) и Программе «Цифровая экономика РФ». При этом специализированных интеллектуальных инструментов для методической работы практически нет: существующие системы (1С:Университет, ТАНДЕМ.Университет, GosInsp) автоматизируют хранение документов, но не поддерживают содержательную работу с компетенциями и нормативами.

Гибридный подход — детерминированные расчёты + LLM-рекомендации — оптимален для данной задачи, так как:
- нормативные расчёты должны воспроизводиться и верифицироваться независимо от ИИ;
- LLM эффективен именно для интерпретации результатов, а не для генерации структурных данных.

«Прорывные» решения в области:
- **GPT-4 + RAG для нормативных документов** (OpenAI, Microsoft Education) — высокое качество, требует внешних API, зависимость от облака.
- **Локальные LLM (Ollama, LM Studio)** — полная изоляция данных, работа без интернета, подходит для вузов с требованиями к персональным данным.
- **Vector DB + RAG (Qdrant, ChromaDB)** — профессиональные системы индексации; в MVP заменены на in-memory numpy для снижения зависимостей.

#### Формулировка и проверка гипотез

**Гипотеза H1**: Детерминированные продукционные правила обеспечивают полный охват нормативных требований ФГОС ВО к структуре учебного плана, не требуя участия LLM в расчётах.

*Результат проверки*: Реализовано 21 правило в `engine.py`, охватывающих все ключевые проверки ФГОС. LLM используется только для текстовых рекомендаций. Гипотеза подтверждена.

**Гипотеза H2**: RAG на основе ФГОС PDF и seed-данных обеспечивает контекстно-релевантные ответы ИИ-ассистента без необходимости дообучения LLM.

*Результат проверки*: Двухпульная RAG-система (ФГОС-документы + данные БД) показала корректное цитирование нормативных положений. Гипотеза подтверждена в условиях наличия PDF ФГОС.

**Гипотеза H3**: Семантический поиск на основе мультиязычной модели эмбеддингов позволяет эффективно сопоставлять запросы на естественном языке с перечнем дисциплин и компетенций.

*Результат проверки*: `paraphrase-multilingual-MiniLM-L12-v2` обеспечивает адекватный поиск по русскоязычным наименованиям дисциплин и описаниям компетенций. Гипотеза подтверждена.

#### Карта ценностного предложения

**Потребительский сегмент — Методист**:
- Работа (Jobs): разработать и проверить УП в соответствии с ФГОС, минимизировать правки при согласовании.
- Боли (Pains): трудоёмкая ручная проверка; риск пропустить нарушения; поиск по ФГОС PDF занимает время.
- Выгоды (Gains): быстрое формирование структуры, автоматическая нормативная проверка, обоснование решений через цитирование ФГОС.

**Ценностное предложение системы**:
- Снижает трудозатраты на первичный подбор дисциплин (Таблица 1 с рекомендациями).
- Исключает арифметические ошибки (автоматический расчёт з.е. и часов).
- Обеспечивает прозрачность нарушений (21 правило с уровнями critical/error/warning).
- Предоставляет ИИ-ассистента для консультаций по нормативам с опорой на актуальные ФГОС PDF.

#### Концепция MVP

MVP ориентирован на методиста одного вуза, одного направления подготовки (бакалавриат, направления 09.03.01–09.03.04), работающего локально. Исключены из MVP: многопользовательский режим, аутентификация, версионирование, экспорт в DOCX/PDF, загрузка PDF через UI.

## ГЛАВА 2. МОДЕЛИРОВАНИЕ И ПРОЕКТИРОВАНИЕ ПРЕДМЕТА ИССЛЕДОВАНИЯ

### 2.1 Функциональное моделирование предметной области

#### Обоснование выбора среды моделирования

Для функционального моделирования используются нотации **IDEF0** (функциональные блоки, декомпозиция) и **DFD** (потоки данных), реализованные в среде **Ramus**. Причины выбора: поддержка обеих нотаций в одном инструменте, возможность последовательной декомпозиции, бесплатная лицензия.

#### Контекстная диаграмма «как есть» (IDEF0 A0)

```
Вход:               [Исходные данные ОП] [Материалы ПООП]
                                ↓
        ┌─────────────────────────────────────────────┐
        │  A0: Формирование ОПОП и учебного плана    │
        └─────────────────────────────────────────────┘
                                ↓
Выход:              [Проект ОПОП] [Проект учебного плана]

Управление: ФГОС ВО, профстандарты, ЛНА вуза, законодательство
Механизмы:  Педагогический и административный персонал
```

Декомпозиция A0 → 6 функций: A1 Выбор компетенций → A2 Определение трудовых функций → A3 Определение перечня дисциплин → A4 Определение объёма → A5 Соответствие компетенций и дисциплин → A6 Конструирование документов.

Критические места модели «как есть»:
- A3 и A5 выполняются вручную на основе экспертной интерпретации.
- Нормативная проверка выполняется на финальном этапе A6, что повышает стоимость исправлений.

#### Контекстная диаграмма «как должно быть» (IDEF0 A0)

```
Вход:               [Перечень компетенций ОПОП] [Документы-источники (ПООП, планы)]
                                ↓
        ┌─────────────────────────────────────────────────────┐
        │  A0: Модуль формирования учебного плана (ИС)       │
        └─────────────────────────────────────────────────────┘
                                ↓
Выход:   [Проект УП] [Отчёт о проверках] [Рекомендации по корректировке]

Управление: Нормативные параметры (ФГОС), правила формирования УП
Механизмы:  Методист, подсистема нормализации, модуль проверок, LLM-модуль
```

Декомпозиция A0 → 5 функций:
- A1 Подготовка рекомендательной базы (ETL из PDF/XLSX → seed JSON → БД)
- A2 Подбор дисциплин и практик по компетенциям (Таблица 1)
- A3 Формирование структуры УП (Таблица 2)
- A4 Автоматические расчёты и проверки (Таблица 3 + движок правил)
- A5 Формирование рекомендаций (LLM + RAG)

#### Диаграмма потоков данных (DFD)

```
[Методист] ──запрос──→ [P1: Подбор дисциплин] ←── D1: База рекомендаций
                              ↓ выбранные элементы
[Методист] ←выбор──── [P2: Формирование структуры УП] ──→ D2: База УП
                              ↓ сохранённая структура
                        [P3: Нормативные проверки] ←── D3: Нормативные параметры
                              ↓ отчёт о нарушениях
                        [P4: LLM-рекомендации] ←── D4: ФГОС PDF chunks (RAG)
                              ↓ текстовые рекомендации
[Методист] ←─────────── [P5: Отображение результатов]
```

Хранилища:
- **D1** — таблица `recommended_elements` (seed из `poop_disciplines.json`)
- **D2** — таблицы `curriculum_plans` + `plan_elements`
- **D3** — таблица `normative_params` (seed из `normative_params.json`)
- **D4** — PDF-файлы `backend/seed/fgosvo/{код}/fgos.pdf` + in-memory chunks

#### Требования к функционалу (реализованные в MVP)

**Функциональные требования:**

1. Создание, хранение и удаление учебных планов (CRUD).
2. Просмотр рекомендованных дисциплин и практик по направлению подготовки (Таблица 1).
3. Предотвращение дублирования обязательных ФГОС-дисциплин при переносе (группировка по `fgos_requirement`).
4. Перенос выбранных элементов в структуру учебного плана (Таблица 2) с upsert-логикой.
5. Редактирование, добавление и удаление элементов УП (Таблица 2).
6. Автоматический расчёт часов из з.е. при каждом сохранении (`hours = credits × 36`).
7. Привязка элементов к компетенциям; семантический подбор компетенций по названию дисциплины.
8. Просмотр агрегированных показателей: по блокам, годам, семестрам, доля обязательной части.
9. Нормативная проверка (21 правило) с классификацией нарушений.
10. Генерация текстовых рекомендаций LLM по результатам проверки.
11. ИИ-ассистент: вопросы об учебном плане и нормативах с RAG-контекстом из ФГОС PDF.
12. Семантический поиск по рекомендованным дисциплинам.
13. Экспорт учебного плана в XLSX.
14. Управление статусом плана: draft → checked → approved.

**Нереализованные (за рамками MVP):**
- Аутентификация и разграничение прав
- Версионирование УП
- Экспорт в DOCX/PDF
- Загрузка PDF через UI

#### Требования к режиму функционирования

- **Интерактивный режим**: пересчёт часов и агрегатов — на каждое сохранение элемента (SQLAlchemy event listener `before_insert`/`before_update`).
- **Пакетный режим**: загрузка seed-данных при старте приложения (функция `load_seed_data` в `main.py`); ETL из PDF — отдельный скрипт `import_poop_pdf.py`.
- **Однопользовательский режим** (MVP): без аутентификации, один узел.

### 2.3 Проектирование интеллектуальной информационной системы

#### Идентификация задач и их характеристик

| Задача | Тип | Метод решения |
|--------|-----|---------------|
| Нормативная проверка УП | Детерминированная | Продукционные правила |
| Подбор дисциплин по компетенциям | Поиск по аналогии | Семантический поиск (эмбеддинги) |
| Объяснение нарушений | Интерпретация | LLM + RAG |
| Ответы на вопросы по нормативам | Вопрос-ответ | RAG + LLM |
| Расчёт агрегатных показателей | Вычислительная | Алгоритмы (SQL + Python) |
| Подбор компетенций для дисциплины | Семантический поиск | Эмбеддинги + cos_sim |

#### Извлечение знаний из эксперта

В ходе проектирования проводились консультации с методистами и преподавателями, выступавшими экспертами. Были установлены:

1. Полный перечень нормативных проверок (21 правило), требования к уровням нарушений.
2. Источники данных для подбора дисциплин: ПООП, планы других вузов, внутренний перечень.
3. Ограничение: для ПКС-компетенций авто-подбор некорректен — требуется экспертная оценка.
4. Формула расчёта часов: 1 з.е. = 36 ч (академические часы).
5. Принцип разделения дисциплин на обязательную и вариативную части.

Модель знаний системы строится на двух механизмах:
- **Онтологическая модель** (разработана в Protégé, OWL) — описывает сущности и связи предметной области (УчебныйПлан → ЭлементУП → Компетенция).
- **Продукционная модель** — нормативные проверки в форме правил «ЕСЛИ — ТО», реализованные в `engine.py`.

#### Выбор стратегии вывода

Применяется **гибридная стратегия**:

1. **Прямой вывод (forward chaining)** — детерминированный контур:
   - Загружаются данные УП из БД.
   - Последовательно применяются 21 продукционное правило.
   - Формируется структурированный отчёт (CheckReport) с классификацией нарушений.

2. **Генеративный вывод** — интеллектуальный контур:
   - Готовый отчёт передаётся в LLM-модуль.
   - LLM генерирует текстовые пояснения и рекомендации.
   - Для ИИ-ассистента: вопрос пользователя → RAG-поиск → контекст + вопрос → LLM.

**Принципиальное ограничение**: LLM не участвует в нормативных расчётах и не изменяет структуру плана. Это обеспечивает воспроизводимость и верифицируемость результатов.

#### Выбор подсистемы объяснения

**Формальная часть**: для каждого нарушения фиксируются:
- `rule_id` — идентификатор правила,
- `level` — критическая ошибка / ошибка / предупреждение,
- `message` — текстовое описание нарушения,
- `actual` — фактическое значение,
- `expected` — нормативное значение.

**Текстовая часть**: `llm_recommendations` в отчёте — пояснения LLM на основе формального отчёта. Формируется промпт через `prompt_builder.py`, генерируется через `OllamaAdapter`.

**RAG-ассистент**: в чате пользователь получает ответы с явными ссылками на источник: `[ФГОС ВО 090301, п. 4.3]`.

#### Выбор подсистемы взаимодействия с пользователем

Web-приложение. Организовано как три последовательных представления:

| Экран | Назначение | Компонент |
|-------|-----------|-----------|
| Главная / список планов | Создание, просмотр, удаление УП | `PlanSetup.jsx` |
| Таблица 1 | Предварительный подбор дисциплин | `Table1.jsx` |

| ИИ-ассистент (кнопка в правом нижнем углу) | Диалог с RAG-ассистентом | `AiChat.jsx` |

#### RAG-конвейер — полное описание

RAG-система состоит из 4 компонентов:

**1. Парсер ФГОС PDF (`fgos_parser.py`)**

Обрабатывает PDF-файлы из `backend/seed/fgosvo/{09.03.0X}/fgos.pdf`:
- Извлекает текст постранично через `fitz.open()`
- Удаляет повторяющиеся заголовки страниц (регулярное выражение `_HEADER_RE`)
- Объединяет строки, разбитые PDF-рендерером (`_join_wrapped_lines`)
- Разбивает текст на чанки по номерам разделов (`1.1.`, `2.4.`, `I.` и т.д.) через `_SECTION_RE`
- Ограничения: минимум 80 символов (`_MIN_CHUNK_LEN`), максимум 1400 символов (`_MAX_CHUNK_LEN`)
- **Кэширование**: раз в сессию на `program_code` (словарь `_fgos_chunk_cache`)

Отображение кодов: `"090301" → "09.03.01"`, `"090302" → "09.03.02"` и т.д.

**2. Генератор чанков из БД (`chunker.py`)**

Строит чанки из структурированных данных БД. Три типа чанков:

*Нормативный чанк (source_type="fgos"):*
```
«Нормативные объёмы блоков программы бакалавриата {code}. 
 Минимальный объём блока 1 — не менее {X_b1} з.е. ...»
```

*Агрегированные компетенции по типу (source_type="fgos"):*
```
«Перечень ОПК (общепрофессиональные компетенции) в ФГОС ВО для направления {code}:
 ОПК-1. Название.
 ОПК-2. Название.»
```

*Индивидуальные компетенции (source_type="fgos"):*
```
«ОПК-1 (ОПК): Название. Описание.»
```

*Рекомендованные элементы (source_type="element"):*
```
«Дисциплина направления {code}: «Название». Блок 1, обязательная часть.
 Кредиты: 4 з.е. Семестры: 3, 4. Источник: ПООП. Формирует компетенции: ОПК-1, УК-2.»
```

**3. Ретривер (`retriever.py`)**

Двухпульная архитектура:
- **ФГОС-пул**: PDF-чанки + компетенция-чанки (70% запрошенных результатов)
- **DB-пул**: чанки рекомендованных дисциплин (30%)

Алгоритм поиска:
1. Строится корпус чанков (кэшируется по `program_code` + MD5-хеш содержимого)
2. Кодируется запрос: `query_vec = embed([query])[0]`
3. Вычисляется косинусная схожесть: `scores = vecs @ query_vec`
4. Отбираются top-K при условии `score ≥ 0.25` (`_MIN_SCORE`)
5. Результаты: сначала ФГОС-пул, затем DB-пул

**Инвалидация кэша**: при изменении содержимого (переименование элемента, изменение нормативов) MD5-хеш меняется → корпус перестраивается автоматически.

**4. Сервис чата (`chat_service.py`)**

Сборка промпта:
```
[RAG-контекст: ФГОС-чанки]        ← первое место (высокое внимание модели)
[Контекст плана: структура, агрегаты, нарушения]
=== ВОПРОС ===
{сообщение пользователя}
=== ВАЖНО ===
Ответь, опираясь на раздел «НОРМАТИВНАЯ БАЗА». Цитируй источники.
```

Такой порядок снижает эффект «потери в середине» (lost-in-the-middle) для небольших моделей.

System-промпт:
```
Ты — методист-помощник по проектированию ОПОП/ФГОС для направления {code}.
Используй ТОЛЬКО предоставленные разделы. Не добавляй знания из памяти.
Отвечай на русском. Будь конкретным. Ссылайся на источник.
```

### 3.7 Руководство пользователя (краткое)

#### Первый запуск

1. Установить Ollama, скачать модель: `ollama pull llama3:latest`
2. Создать виртуальное окружение: `python -m venv .venv && .venv\Scripts\activate`
3. Установить зависимости: `pip install -r requirements.txt`
4. Установить frontend: `cd frontend && npm install && cd ..`
5. Положить PDF ФГОС ВО в `backend/seed/fgosvo/09.03.0X/fgos.pdf`
6. Запустить backend: `uvicorn backend.main:app --reload`
7. Запустить frontend: `cd frontend && npm run dev`
8. Открыть `http://localhost:5173`

#### Сценарий работы

**Шаг 1. Создание учебного плана**
- Открыть главную страницу → «Создать план»
- Указать название и направление подготовки (09.03.01–09.03.04)

**Шаг 2. Таблица 1 — предварительный подбор**
- Просмотреть рекомендованные дисциплины и практики из ПООП и лучших практик
- Отметить нужные элементы (обязательные отмечены автоматически и заблокированы)
- Использовать семантический поиск для нахождения дисциплин по смыслу запроса
- Нажать «Перенести в учебный план»

**Шаг 3. Таблица 2 — редактирование структуры**
- Добавлять, изменять, удалять дисциплины, практики, ГИА, факультативы
- Для каждого элемента: название, блок, часть, з.е., семестры, компетенции
- Часы рассчитываются автоматически (з.е. × 36)
- Использовать «Подобрать компетенции» для семантического подбора

**Шаг 4. Таблица 3 — проверка и отчёт**
- Нажать «Запустить проверку» — выполняются 21 правило
- Просмотреть нарушения: critical (красный), error (оранжевый), warning (жёлтый)
- Прочитать рекомендации LLM
- Устранить нарушения → вернуться в Таблицу 2

**Шаг 5. Утверждение и экспорт**
- При отсутствии critical/error — сменить статус на «approved»
- Скачать XLSX через кнопку экспорта

**Шаг 6. ИИ-ассистент**
- Открыть через кнопку в правом нижнем углу
- Задавать вопросы: «Каков минимальный объём Блока 1?», «Какие ОПК определены ФГОС?»
- Ответы цитируют конкретные пункты ФГОС при наличии PDF-файлов

### 4.4 Потенциальные риски. Карта рисков

| Риск | Вероятность | Влияние | Оценка | Стратегия |

### 4.5 SWOT-анализ

|  | Возможности | Угрозы |
|--|------------|--------|
| **Сильные стороны** | **SO**: Использовать уникальность ИИ+RAG для занятия ниши «умная проверка ФГОС» ещё до прихода крупных игроков | **ST**: Открытый код и on-premise развёртывание как ответ на угрозу ИБ-ограничений вузов |
| **Слабые стороны** | **WO**: Устранить ограничения MVP (многопользовательский режим, DOCX-экспорт) и выйти на рынок с полноценным продуктом | **WT**: Установить партнёрства с вузами для совместного финансирования доработки системы |

**Сильные стороны (S)**:
- Полный нормативный контур: 21 правило, все ключевые требования ФГОС
- Локальная работа без интернета и передачи данных на сторонние сервера
- Открытый исходный код — доверие и возможность адаптации
- Семантический поиск + RAG: уникально для данного сегмента

**Слабые стороны (W)**:
- Нет аутентификации и разграничения прав (MVP)
- Нет версионирования документов
- Зависимость от Ollama и качества модели llama3
- Высокие требования к железу (8 ГБ RAM)

**Возможности (O)**:
- Масштабирование на все направления подготовки (сейчас только 09.03.x)
- Замена SQLite на PostgreSQL для многопользовательской работы
- Интеграция с ЕСИА и вузовскими ИС (1С, ТАНДЕМ)
- Государственная поддержка ИИ-инструментов для образования

**Угрозы (T)**:
- Изменения ФГОС ВО 3++ — требуют обновления правил и seed-данных
- Выход крупных игроков (Сбер ИИ, VK Education) в нишу методических инструментов
- Ограничения ИБ в вузах (запрет на LLM-сервисы)

## ЗАКЛЮЧЕНИЕ

### Итоги реализации

В рамках работы разработан и реализован MVP модуля формирования учебного плана системы «Интеллектуальный методист». Система охватывает полный цикл работы с учебным планом:

**Реализованные функциональные возможности:**
- Три рабочих таблицы в соответствии с требованиями к модулю УП
- 21 продукционное правило нормативного контроля с классификацией critical/error/warning
- Двухпульная RAG-архитектура (ФГОС PDF + данные БД) для ИИ-ассистента
- Семантический поиск на мультиязычной модели эмбеддингов
- Полностью локальная работа без передачи данных во внешние сервисы
- REST API с Swagger-документацией

**Подтверждённые гипотезы:**
1. Детерминированные правила обеспечивают полный нормативный контроль без участия LLM в расчётах.
2. RAG на основе ФГОС PDF + seed-данных обеспечивает корректное цитирование нормативов.
3. Мультиязычные эмбеддинги эффективны для семантического поиска по русскоязычным дисциплинам.

**Практическая значимость:**
Система снижает трудоёмкость первичного подбора дисциплин, исключает арифметические ошибки в расчётах трудоёмкости, обеспечивает прозрачность нормативных проверок и предоставляет методисту интерактивный доступ к требованиям ФГОС на естественном языке.
